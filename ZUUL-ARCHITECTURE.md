# Zuul CI Architecture

## Overview

Zuul is a project gating system that runs Ansible playbooks on provisioned nodes.
This document explains how the components connect, where configuration lives, and
how Ansible inventory and playbooks work in this setup.

---

## Repository Structure

```
victorhugojt/zuul-config          ← config-project (central infrastructure)
victorhugojt/recordtec            ← untrusted project (backend)
victorhugojt/recordtec-fe         ← untrusted project (frontend)
```

### What each repo owns

| Repo | Owns |
|---|---|
| `zuul-config` | Pipelines, base job, node definition |
| `recordtec` | Its own jobs, playbooks, project stanza |
| `recordtec-fe` | Its own jobs, playbooks, project stanza |

## Configuration Files

### `zuul-config/.zuul.yaml` — shared infrastructure only

```yaml
# Pipelines: define when and how CI runs
- pipeline:
    name: check          # triggers on every PR
    ...

- pipeline:
    name: gate           # triggers on PR approval, auto-merges
    ...

# Base job: defines the shared node and common setup/teardown
# Only config-projects can define jobs with parent: null
- job:
    name: base
    parent: null
    run: playbooks/base.yaml           # runs before every job
    post-run: playbooks/post-base.yaml # runs after every job (always)
    nodeset:
      nodes:
        - name: controller             # node name used in Ansible inventory
          label: linux-executor        # nodepool label to request
```

### `recordtec/.zuul.yaml` — backend repo owns its CI

```yaml
# Jobs: define what to run (playbooks live in THIS repo)
- job:
    name: recordtec-lint
    parent: base                       # inherits node + setup from base
    run: playbooks/lint.yaml           # playbooks/lint.yaml inside recordtec repo

- job:
    name: recordtec-unit-tests
    parent: base
    run: playbooks/unit-tests.yaml

- job:
    name: recordtec-integration-tests
    parent: base
    run: playbooks/integration-tests.yaml
    timeout: 3600

- job:
    name: recordtec-build
    parent: base
    run: playbooks/build.yaml

# Project stanza: maps jobs to pipelines
- project:
    merge-mode: squash
    check:
      jobs:
        - recordtec-lint
        - recordtec-unit-tests
        - recordtec-integration-tests
        - recordtec-build
    gate:
      jobs:
        - recordtec-lint
        - recordtec-unit-tests
        - recordtec-integration-tests
        - recordtec-build
```

### `recordtec-fe/.zuul.yaml` — frontend repo owns its CI

```yaml
- job:
    name: recordtec-fe-lint
    parent: base
    run: playbooks/lint.yaml

- job:
    name: recordtec-fe-unit-tests
    parent: base
    run: playbooks/unit-tests.yaml

- job:
    name: recordtec-fe-e2e-tests
    parent: base
    run: playbooks/e2e-tests.yaml
    timeout: 3600

- job:
    name: recordtec-fe-build
    parent: base
    run: playbooks/build.yaml

- project:
    merge-mode: squash
    check:
      jobs:
        - recordtec-fe-lint
        - recordtec-fe-unit-tests

        - recordtec-fe-build
    gate:

        - recordtec-fe-lint

        - recordtec-fe-build
```

## Ansible Inventory — Auto-generated by Zuul

You never write an inventory file. Zuul generates it automatically from the
`nodeset` definition in the base job.

Given this nodeset:
```yaml
nodeset:
  nodes:
    - name: controller       # ← this becomes the Ansible group name
      label: linux-executor  # ← nodepool provisions a node with this label
```

Zuul generates an inventory equivalent to:
```ini
[controller]
172.18.0.8 ansible_user=zuul ansible_python_interpreter=/usr/bin/python3
```

This is why every playbook targets `hosts: controller` — it matches the node
name defined in the base job's nodeset.

## Execution Flow

When a PR is opened in `recordtec`, Zuul runs each job in this sequence:

```
1. GitHub sends webhook → Zuul scheduler receives PR event
       ↓
2. Zuul matches event → check pipeline → recordtec jobs
       ↓
3. Nodepool provisions a node (zuul-node container, label: linux-executor)
       ↓
4. Zuul executor SSHes into the node as user "zuul"
       ↓
5. Zuul checks out code on the node:
     /home/zuul/src/github.com/victorhugojt/zuul-config/   ← config-project
     /home/zuul/src/github.com/victorhugojt/recordtec/     ← project under test
       ↓
6. Executor runs Ansible playbooks in order:
     base.yaml          (from zuul-config) — setup
     lint.yaml          (from recordtec)   — actual CI step
     post-base.yaml     (from zuul-config) — teardown (always runs)
       ↓
7. Results reported back to GitHub as Check Runs
```

## Zuul Variables Available in Every Playbook

Zuul injects these variables into every Ansible playbook automatically:

| Variable | Example value | Description |
|---|---|---|
| `zuul.project.src_dir` | `/home/zuul/src/github.com/victorhugojt/recordtec` | Absolute path to the checked-out project |
| `zuul.project.name` | `victorhugojt/recordtec` | Full project name |
| `zuul.project.short_name` | `recordtec` | Repo name only |
| `zuul.branch` | `main` | Target branch |
| `zuul.pipeline` | `check` | Pipeline that triggered the job |
| `zuul.change` | `42` | PR number |

## Example Playbooks

### `zuul-config/playbooks/base.yaml` — common setup

```yaml
---
- hosts: controller
  tasks:
    - name: Show workspace
      command: pwd
      args:
        chdir: "{{ zuul.project.src_dir }}"
```

### `zuul-config/playbooks/post-base.yaml` — common teardown

```yaml
---
- hosts: controller
  tasks:
    - name: Job completed
      debug:
        msg: "Job finished on {{ inventory_hostname }}"
```

### `recordtec/playbooks/lint.yaml` — project-specific

```yaml
---
- hosts: controller
  tasks:
    - name: Install linting tools
      pip:
        name: [flake8, pylint]

    - name: Run flake8
      command: flake8 src/
      args:
        chdir: "{{ zuul.project.src_dir }}"

    - name: Run pylint
      command: pylint src/
      args:
        chdir: "{{ zuul.project.src_dir }}"
```

### `recordtec-fe/playbooks/lint.yaml` — project-specific

```yaml
---
- hosts: controller
  tasks:
    - name: Install npm dependencies
      command: npm ci
      args:
        chdir: "{{ zuul.project.src_dir }}"

    - name: Run eslint
      command: npm run lint
      args:
        chdir: "{{ zuul.project.src_dir }}"
```

## Key Rules

- **Pipelines** can only be defined in a `config-project`
- **`parent: null`** can only be set in a `config-project`
- **Untrusted projects** can define jobs, but they must inherit from a config-project job
- **Playbook paths** in a job are relative to the repo where the job is defined
- **Inventory** is always auto-generated by Zuul — never write your own
- **Node name** in `nodeset` must match `hosts:` in your playbooks
